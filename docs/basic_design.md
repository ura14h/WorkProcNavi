# 作業手順ナビ / WorkProcNavi 基本設計

## 1. 文書の目的

本書は、手順書に沿った作業をフェーズ単位でガイドし、確認しながら完了まで導くローカルアプリ
「作業手順ナビ / WorkProcNavi」の基本設計をまとめるものである。

本書の目的は以下の通り。

- アプリの対象課題と提供価値を明確にする
- 本バージョンで実装対象とする範囲を定義する
- 主要な画面、機能、データ構造を整理する
- マルチプラットフォーム対応を前提とした技術方針を決める
- 詳細設計と実装に進むための土台を作る

## 2. 背景と課題

現場作業や定型業務では、既存の手順書があっても以下の課題が発生しやすい。

- 手順書が文書のままで、今どこまで進んだか分かりにくい
- フェーズごとの完了条件が曖昧で、確認漏れが起きる
- 作業者が次に何をすべきか迷いやすい
- 中断後の再開時に、どこから戻ればよいか分かりにくい
- 手順の実施記録が残りにくい

WorkProcNavi は、既存の手順書を「実行可能なウィザード」として扱い、作業者が段階的に確認しながら安全に完了まで進められる体験を提供する。

加えて、手順書をアプリ専用データとして重く管理するのではなく、単一の Markdown ファイル、または Markdown と画像群をまとめた ZIP パッケージをそのまま投入してすぐ開始できる、軽量なローカルアプリとして設計する。

## 3. 目標

### 3.1 プロダクト目標

- 手順書に沿って、迷わず順番に作業を進められること
- 各フェーズで確認を行い、確認漏れを減らすこと
- 作業進捗を保存し、中断と再開を容易にすること
- 小さなローカルアプリとして、ネットワーク接続なしでも利用できること
- 同じ仕組みで複数の手順書を扱えること
- 手順完了後に確認結果をエビデンスとしてファイル出力できること

### 3.2 本バージョンの到達点

本バージョンでは以下を満たすことを目標とする。

- 1 つ以上の手順書を読み込める
- 単一の Markdown ファイルまたは ZIP ファイルのドラッグ＆ドロップで手順を開始できる
- セッションファイルのドラッグ＆ドロップで作業を再開できる
- フェーズ単位で手順を表示できる
- 各確認項目のチェックを付けられる
- フェーズ完了時に次へ進める
- 作業状態をセッションファイルとして保存し、再開できる
- 作業完了時にエビデンスファイルを出力できる

## 4. 想定利用シナリオ

### 4.1 基本シナリオ

1. 利用者がアプリを起動する
2. 利用者が単一の Markdown ファイル、または Markdown と画像群を含む ZIP 手順書パッケージをウィンドウへドラッグ＆ドロップする
3. アプリが入力ファイルを読み取り、手順書の概要、フェーズ数、ステップ数を表示する
4. 利用者が作業開始する
5. アプリが現在フェーズの説明と手順を案内する
6. 利用者がフェーズ内の手順を確認し、各確認項目のチェックを行う
7. 現在フェーズ内の確認項目がすべて完了すると、フェーズ完了として次フェーズへ進める
8. 中断時は現在位置を保存する
9. アプリは手順書と同じ場所にセッションファイルを保存する
10. 再開時は利用者がそのセッションファイルをウィンドウへドラッグ＆ドロップし、前回位置から続ける
11. 最終フェーズ完了後、アプリが確認結果をエビデンスファイルとして出力する
12. 利用者が出力ファイルを保存または参照する

### 4.2 利用者像

- 定型作業を手順書に従って実施する作業者
- 作業品質を均一化したい現場リーダー
- PC 上で簡易な業務支援ツールを使いたい小規模チーム

- 手順本文内リンクの安全な外部オープン
- 手順本文リンクの URI ホバー表示
- フェーズ完了判定

- 多言語対応

## 6. 機能要件

### 6.1 手順書管理

- 手順書はローカルファイルとして保持する
- 初期版の入力形式は単一 Markdown ファイル、または Markdown ファイルと任意アセットを含む ZIP パッケージとする
- 利用者は `.md`、`.zip`、またはセッションファイルをドラッグ＆ドロップして開始または再開できる
- アプリは入力ファイルを解析し、内部の手順モデルへ変換して実行する
- `.md` と `.zip` はどちらも同じ開始フローで扱う
- ZIP パッケージは Markdown ファイル 1 つと、必要に応じて画像などの関連ファイルを含められる
- ZIP パッケージはアプリ内の一時フォルダに展開して扱う
- 手順書の構造は `# 作業名`、`## フェーズ`、`### ステップ`、チェックボックス付き箇条書きの確認項目で表現する
- 第二見出しをフェーズ、第三見出しをステップ、チェックボックス型箇条書きを確認項目として抽出する
- これら以外の本文は、手順を説明する記述として扱う
- Markdown の標準コードブロックを手順本文内で利用できること
- 画像は Markdown 内から相対参照で利用できること
- Markdown の標準リンク記法を手順本文内で利用できること
- 手順書またはセッションファイルの取り込み時に不都合がある場合は、その時点でエラーとして開始しない
- セッションファイルがドロップされた場合は、参照先の手順書を解決して再開する

### 6.2 作業ガイド

- ファイル読込後に概要画面を表示する
- 現在フェーズの内容を HTML ライクにまとめてレンダリングする
- HTML ライクなレンダリングに用いるスタイルシートは、初期版ではアプリ内蔵とする
- 各フェーズについて、見出し配下の説明文、ステップ、確認項目、画像を一続きの文書として表示する
- コードブロックは整形表示し、利用者がワンクリックでクリップボードへコピーできること
- フェーズ内のステップへは、`前のステップ` / `次のステップ` のスクロールジャンプボタンで移動できること
- 現在のフェーズは、画面上部のプログレスバーで分かること
- フェーズ間移動は、ウィザード風の `戻る` / `次へ` ボタンで行うこと
- 現在フェーズ内に未確認項目が残っている場合は確認ダイアログを表示し、利用者が承知した場合のみ次フェーズに進めるようにする

### 6.3 進捗管理

- 現在の手順書 ID、フェーズ位置、ステップ位置、チェック状態、開始時刻をセッションファイルへ保存する
- セッションファイルは元の手順書と同じ場所へ保存する
- セッションファイル名は元ファイル名に `.session` を付けた識別しやすい規則とする
- セッションファイルのバージョンはアプリのバージョンと同じ値を記録する
- 元の入力ファイルの識別情報を保持する
- アプリ再起動後は、セッションファイルをドロップすることで前回セッションを再開できる
- ステップ単位で完了状態を保持する
- 手順書完了時に完了日時を保存する
- 一時停止時に確実に保存し、必要に応じて操作中も同じセッションファイルを更新する
- セッションファイルには解析結果のキャッシュを持たせない
- 作業完了後はセッションファイルを不要とし、エビデンス出力成功後に削除する
- ZIP の一時展開フォルダは、作業完了時またはアプリ終了時に削除する

### 6.4 記録

- 作業完了時に実施結果のサマリーを表示する
- 作業完了時にエビデンスファイルを出力できる
- エビデンスには、手順書名、実施日時、フェーズ/ステップ/確認項目ごとの結果を含める
- エビデンスの章構成は、元の手順書の見出し構造に可能な限り沿わせる

### 6.5 エビデンス出力

- 初期版のエビデンス出力形式は Markdown のみとする
- エビデンスファイル名は元ファイル名に `.log` を付けた識別しやすい規則とする
- 出力ファイルは元の手順書と同じ章構成を基本とし、`# 作業名`、`## フェーズ`、`### ステップ` の順で結果を並べる
- 文書冒頭のメタ情報は YAML front matter とし、開始日時、完了日時、参照した手順書ファイル名を含める
- 各確認項目について、完了状態が分かるチェック付き箇条書きで出力する

### 6.6 操作性

- キーボードとマウスの両方で基本操作が可能であること
- 画面サイズが変わっても主要操作が隠れないこと
- フェーズ、現在位置、残件数が常に分かること
- ドラッグ＆ドロップを主操作として、数クリック以内で開始できること
- 小さなウィンドウでも主要情報が破綻しないこと
- コードブロックのコピー操作が明確で、誤タップしにくいこと
- 作業対象アプリと並べて使えるよう、表示フットプリントを小さく保つこと
- 縦長ウィンドウ、横長ウィンドウのどちらでも使いやすいこと
- 主要表示はフェーズ文書のレンダリングを中心にし、補助 UI を最小限に抑えること

### 6.7 手順本文内リンク

- 手順本文に書かれた Markdown リンクは、アプリ内画面を遷移させず、OS 側の標準機能で開く
- `http://` および `https://` の外部サイト URL は、利用者環境で既定に設定されている外部ブラウザで開く
- `file://` URL は、利用者環境の標準ファイルエクスプローラまたは Finder で開く
- `file://` URL がフォルダを指す場合は、そのフォルダをそのまま開く
- `file://` URL がファイルを指す場合は、ファイル本体を直接開かず、親フォルダを開いて対象ファイルを選択または表示する
- `file://` URL がファイルを指し、対象ファイルが見つからない場合は、見つからないことを利用者へ伝えたうえで、親フォルダだけを開くか確認する
- `file://` URL のパスは、macOS / Linux の `/` 区切り、Windows のドライブ文字や UNC パスを考慮して OS ごとに正規化する
- `javascript:`、`data:`、独自スキームなど、許可していないプロトコルは開かず、利用者に開けないリンクとして通知する
- 手順本文リンクにマウスカーソルを合わせた場合は、約 0.5 秒後に URI をホバー表示する
- URI のホバー表示はリンク上からカーソルが外れた時点で消える
- リンクを開く処理は作業進捗やチェック状態を変更しない

## 7. 非機能要件

### 7.1 対応プラットフォーム

- macOS
- Windows
- Linux

### 7.2 動作方針

- オフラインで動作するローカルアプリとする
- 軽量で、一般的な業務 PC で快適に動作すること
- 手順書や進捗データは端末内に保存する
- 初回起動後すぐ使えることを重視し、常駐機能や重い初期設定を持たない
- 作業対象アプリの横や上下に配置する補助ウィンドウとして使えることを前提とする
- 表示スタイルはアプリ内蔵とし、手順書ごとの追加スタイル読み込みを初期版では行わない
- 初期版の表示言語は日本語のみとする

### 7.3 信頼性

- 確認項目チェックの変更は即時保存する
- 一時停止後にアプリが終了しても、セッションファイルから再開できること
- アプリ異常終了時も、直前の進捗を可能な限りセッションファイルから復元できること
- 手順書またはセッションファイルの取り込み時に不都合がある場合は、開始前にエラーメッセージを表示する
- 初期版ではセッションファイルのバージョン互換性チェックは行わない
- クリップボードコピー失敗時は、その場で利用者に通知すること

### 7.4 保守性

- 手順書データ形式と UI ロジックを分離する
- 将来的に手順書編集機能や承認機能を追加しやすい構造とする

## 8. 画面設計方針

### 8.1 主要画面

1. ホーム画面
   `.md` / `.zip` / セッションファイルのドロップ領域を表示する
2. 手順書詳細画面
   手順書の概要、フェーズ一覧、ステップ数、開始ボタンを表示する
3. 作業実行画面
   上部プログレスバー、現在フェーズの文書レンダリング、手順本文内リンク、リンク URI のホバー表示、`前のステップ` / `次のステップ` ボタン、コードブロックのコピーボタン、フェーズの `戻る` / `次へ` ボタン、および左側の `保存して中断` ボタンを表示する
4. 完了サマリー画面
   実施結果、所要時間、ローカル時刻の開始/完了日時、エビデンス出力先表示、出力先を開く導線を表示する

### 8.2 レイアウト方針

- 主要領域は現在フェーズの文書表示とし、見出し、本文、確認項目、画像、コードブロックをそのまま読めるようにする
- 画面上部にはフェーズ数に対応したプログレスバーを配置し、現在位置を常に把握できるようにする
- 画面下部または末尾には左側に `保存して中断` ボタン、右側にフェーズ移動用の `戻る` / `次へ` ボタンを配置する
- `保存して中断` 実行後は保存済み状態でホーム画面へ戻す
- フェーズ内ステップ移動は、ステップ単位のスクロールジャンプボタンで行う
- 画像やコードブロックはウィンドウ幅に収まるよう縮小または横スクロールで扱う

### 8.3 UI の基本原則

- 1 画面 1 目的を意識し、判断に迷わせない
- 「今どこか」「次に何をするか」を大きく表示する
- 未完了の確認項目が一目で分かるようにする
- 長文手順書でも読みやすい余白と階層を保つ
- 起動直後の第一操作は `.md` / `.zip` / セッションファイルのドロップで完結できるようにする

## 9. データ設計

### 9.1 手順書入力形式

初期版では、単一 Markdown ファイル、または以下のような ZIP パッケージを入力形式とする。

#### 単一 Markdown ファイル

```text
sample-proc-001.md
```

#### ZIP パッケージ

```text
sample-proc-001.zip
  procedure.md
  assets/
    image-01.png
    image-02.jpg
```

#### セッションファイル

```text
sample-proc-001.session
```

セッションファイルの内容は JSON とする。

#### Markdown の想定ルール

- `#` を手順書タイトル
- `##` をフェーズ
- `###` をステップ
- `###` 配下の `- [ ]` を確認項目とする
- これら以外の通常段落やその他の Markdown 記述は、手順を説明する本文として扱う
- コード例やコマンドは Markdown 標準のコードブロックで記述できる
- 画像は Markdown 標準記法で相対参照する
- 外部サイトやローカル参照先は Markdown 標準リンク記法で記述できる
- リンクとして開く対象は `http://`、`https://`、`file://` を初期版の対象とする
- 初期版の ZIP には Markdown ファイルを 1 つだけ含める
- 初期版では CSS などの外部スタイル定義ファイルは入力対象に含めない
- 初期版の表示言語は日本語のみとする

#### Markdown 記述例

```md
# 機器セットアップ手順

## 事前確認

事前に必要物品と対象端末の状態を確認する。

### 電源状態を確認する

対象端末の電源が OFF であることを確認する。

- [ ] 端末の電源ランプが消灯している
- [ ] 電源ケーブルの状態に問題がない

### 付属品を確認する

[メーカーのサポートサイト](https://example.com/support) を確認し、必要な更新情報がないことを確認する。

- [ ] AC アダプタがそろっている
- [ ] 必要な周辺機器がそろっている
```

外部入力は Markdown ベースとするが、アプリ内部では表示・進捗管理のため扱いやすい手順構造へ変換する。

### 9.2 内部処理方針

アプリ内部では、取り込んだ Markdown を以下の観点で正規化して扱う。

- フェーズ、ステップ、確認項目の階層を明示的に扱えること
- 画像、コードブロック、本文をレンダリングしやすい形に扱えること
- フェーズ位置、ステップ位置、チェック状態と結びつけられること

内部で使用する具体的なデータモデルや型定義は、詳細設計で定義する。

### 9.3 セッションファイル形式

```json
{
  "sessionId": "session-1712202000000",
  "appVersion": "1.2.0",
  "manualId": "sample-proc-001",
  "sourcePath": "/path/to/sample-proc-001.zip",
  "sourceType": "zip",
  "sessionFilePath": "/path/to/sample-proc-001.session",
  "status": "in_progress",
  "currentPhaseId": "phase-001",
  "currentStepId": "phase-001-step-002",
  "checkedItemIds": [
    "phase-001-step-001-check-001"
  ],
  "phaseTransitionRecords": [
    {
      "phaseId": "phase-001",
      "leftAt": "2026-04-04T09:12:00+09:00",
      "advanceMode": "forced_with_pending",
      "pendingConfirmItemIds": [
        "phase-001-step-001-check-002"
      ]
    }
  ],
  "startedAt": "2026-04-04T09:00:00+09:00",
  "updatedAt": "2026-04-04T09:10:00+09:00",
  "completedAt": null
}
```

セッションファイルは元の手順書と同じディレクトリに保存し、元ファイル名から識別しやすい命名とする。
セッションファイルの `appVersion` には、保存時点のアプリバージョンをそのまま記録する。
初期版では、この値による互換性チェックは行わない。

作業完了後は不要となるため、エビデンス出力成功後に削除する。

例:

```text
sample-proc-001.md
sample-proc-001.log
```

```text
sample-proc-001.zip
sample-proc-001.log
```

### 9.4 エビデンス出力形式

エビデンスは `.log` ファイルとして出力し、内容は Markdown で、元の手順書に近い構成で確認結果を残す。

```md
---
実施開始: "2026-04-04T09:00:00+09:00"
実施完了: "2026-04-04T09:35:00+09:00"
手順書名: "sample-proc-001.zip"
---

# 機器セットアップ手順 実施記録

## 事前確認

### 電源状態を確認する

- [x] 端末の電源ランプが消灯している
- [x] 電源ケーブルの状態に問題がない
```

## 10. 実装方針

### 10.1 現時点の方針

- 手順書の入力形式は単一 Markdown ファイル、または ZIP パッケージとする
- 再開入力としてセッションファイルを受け付ける
- ZIP 内の Markdown ファイルを読み込み、内部モデルへ変換する
- Markdown の見出しとチェックボックスだけで手順構造を表現する
- `.md` と `.zip` は同じ開始フローで処理する
- ZIP はアプリ内の一時フォルダへ展開して処理し、完了時またはアプリ終了時に削除する
- 表示用スタイルシートはアプリ内蔵とする
- 手順本文内リンクはアプリ内で直接遷移せず、外部サイト URL は既定ブラウザ、`file://` は標準ファイルエクスプローラまたは Finder へ委譲する
- `file://` がファイルを指す場合は、ファイルを直接起動せず親フォルダ内で対象ファイルを表示する
- 対象ファイルが見つからない場合は、親フォルダを開くかを確認してから続行する
- 進捗保存は元手順書の横に置く `.session` ファイル、エビデンス出力は `.log` ファイルで開始する
- セッションファイルにはキャッシュを持たせない
- アプリ内のクローズドな履歴管理は持たない

### 10.2 技術選定の扱い

- 開発プラットフォーム、UI フレームワーク、言語、保存ライブラリなどの技術選定は本書では確定しない
- 先に入出力仕様と操作要件を固め、その後に実装方式を選定する
- 技術選定時は、マルチプラットフォーム対応、軽量性、リサイズ性、ローカルファイル操作のしやすさを評価軸とする

## 11. 想定成果物

実装時に最低限必要となる成果物のイメージは以下。

```text
docs/
  basic_design.md
resources/

manuals/
  sample-proc-001.md
  sample-proc-001.zip
  sample-proc-001.session
  sample-proc-001.log
```

`.session` は作業中のみ存在し、完了後は削除される前提とする。

## 13. 実装フェーズ案

### Phase 1: 手順書表示の土台

- アプリ起動
- `.md` / `.zip` / セッションファイル ドロップ受付
- Markdown 解析
- 手順書概要表示

### Phase 2: 作業実行

- フェーズ表示
- 確認項目チェック
- コードブロックコピー
- 手順本文内リンクの安全な外部オープン
- 次フェーズ遷移

### Phase 3: 保存と再開

- セッションファイル保存
- 中断/再開
- 完了サマリー
- エビデンス出力

### Phase 4: 改善

- 手順書パッケージ作成支援

## 14. 現時点の提案

本バージョンでは、「単一 Markdown、Markdown と画像をまとめた ZIP パッケージ、または手順書の横に保存されたセッションファイルをドロップすると、そのまま開始または再開でき、完了時にエビデンスを出力するローカルデスクトップアプリ」を実装対象として定義する。

この方針であれば、仕様が過度に広がらず、以下の価値を早く確認できる。

- 手順書の作成と配布が現実的に回るか
- ドロップしてすぐ始める体験が十分に軽快か
- セッションファイル方式で再開運用が自然に回るか
- 実際に作業ガイドとして使いやすいか
- フェーズ確認の体験が妥当か
- 現場で必要な記録粒度が十分か
- エビデンス出力が運用に耐えるか
