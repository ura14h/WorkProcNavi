# 作業手順ナビ / WorkProcNavi 基本設計

## 1. 文書の目的

本書は、手順書に沿った作業をフェーズ単位でガイドし、確認しながら完了まで導くローカルアプリ
「作業手順ナビ / WorkProcNavi」の基本設計をまとめるものである。

本書の目的は以下の通り。

- アプリの対象課題と提供価値を明確にする
- MVP の実装範囲を定義する
- 主要な画面、機能、データ構造を整理する
- マルチプラットフォーム対応を前提とした技術方針を決める
- 詳細設計と実装に進むための土台を作る

## 2. 背景と課題

現場作業や定型業務では、既存の手順書があっても以下の課題が発生しやすい。

- 手順書が文書のままで、今どこまで進んだか分かりにくい
- フェーズごとの完了条件が曖昧で、確認漏れが起きる
- 作業者が次に何をすべきか迷いやすい
- 中断後の再開時に、どこから戻ればよいか分かりにくい
- 手順の実施記録が残りにくい

WorkProcNavi は、既存の手順書を「実行可能なウィザード」として扱い、作業者が段階的に確認しながら安全に完了まで進められる体験を提供する。

## 3. 目標

### 3.1 プロダクト目標

- 手順書に沿って、迷わず順番に作業を進められること
- 各フェーズで確認を行い、確認漏れを減らすこと
- 作業進捗を保存し、中断と再開を容易にすること
- 小さなローカルアプリとして、ネットワーク接続なしでも利用できること
- 同じ仕組みで複数の手順書を扱えること

### 3.2 MVP 目標

初期版では以下を満たすことを目標とする。

- 1 つ以上の手順書を読み込める
- フェーズと手順を順に表示できる
- 各手順の確認チェックを付けられる
- フェーズ完了時に次へ進める
- 作業状態をローカル保存し、再開できる

## 4. 想定利用シナリオ

### 4.1 基本シナリオ

1. 利用者がアプリを起動する
2. 利用者が対象の手順書を選択する
3. アプリが手順書の概要、所要時間、注意事項を表示する
4. 利用者が作業開始する
5. アプリが現在フェーズの説明と手順を 1 つずつ、または一覧で案内する
6. 利用者が各手順の実施後に確認チェックを行う
7. 必須確認がすべて完了すると、フェーズ完了として次フェーズへ進める
8. 必要に応じてメモを残す
9. 中断時は現在位置を保存する
10. 再開時は前回位置から続ける
11. 最終フェーズ完了後、作業完了として記録を残す

### 4.2 利用者像

- 定型作業を手順書に従って実施する作業者
- 作業品質を均一化したい現場リーダー
- PC 上で簡易な業務支援ツールを使いたい小規模チーム

## 5. 対象範囲

### 5.1 MVP に含む

- 手順書のローカル読込
- 手順書一覧表示
- 手順書詳細表示
- フェーズごとの進行ガイド
- 手順チェック
- フェーズ完了判定
- 作業進捗の自動保存
- 中断と再開
- 完了記録の保存
- 日本語 UI

### 5.2 MVP では含めない

- 複数ユーザー管理
- クラウド同期
- リアルタイム共同作業
- 手順書の高度な編集機能
- 外部システム連携
- 写真や動画のアップロード管理
- 承認ワークフロー

## 6. 機能要件

### 6.1 手順書管理

- 手順書はローカルファイルとして保持する
- アプリは指定フォルダ、またはアプリ内の既定フォルダから手順書を読み込む
- 手順書にはタイトル、版数、説明、フェーズ、手順、注意事項を持たせる
- 初期版では JSON を標準形式とする

### 6.2 作業ガイド

- 手順書選択後に概要画面を表示する
- 各フェーズについて、目的、注意点、完了条件を表示する
- 各手順について、指示内容、補足、確認チェックを表示する
- 必須手順が完了するまで、次フェーズに進めないようにする
- フェーズ切替時に確認ダイアログを表示する

### 6.3 進捗管理

- 現在の手順書 ID、フェーズ位置、チェック状態、メモ、開始時刻をローカル保存する
- アプリ再起動後に前回セッションを再開できる
- 手順単位で完了状態を保持する
- 手順書完了時に完了日時を保存する

### 6.4 記録

- 各手順に対して任意メモを残せる
- 作業完了時に実施結果のサマリーを表示する
- 完了済みセッションを一覧で参照できる

### 6.5 操作性

- キーボードとマウスの両方で基本操作が可能であること
- 画面サイズが変わっても主要操作が隠れないこと
- フェーズ、現在位置、残件数が常に分かること

## 7. 非機能要件

### 7.1 対応プラットフォーム

- macOS
- Windows
- Linux

### 7.2 動作方針

- オフラインで動作するローカルアプリとする
- 軽量で、一般的な業務 PC で快適に動作すること
- 手順書や進捗データは端末内に保存する

### 7.3 信頼性

- 手順チェックの変更は即時保存する
- アプリ異常終了時も、直前の進捗を可能な限り復元できること
- 手順書フォーマット不正時は、原因が分かるエラーメッセージを表示する

### 7.4 保守性

- 手順書データ形式と UI ロジックを分離する
- 将来的に手順書編集機能や承認機能を追加しやすい構造とする

## 8. 画面設計方針

### 8.1 主要画面

1. ホーム画面
   利用可能な手順書一覧、最近の作業、再開導線を表示する
2. 手順書詳細画面
   手順書の説明、版数、フェーズ一覧、注意事項、開始ボタンを表示する
3. 作業実行画面
   現在フェーズ、手順一覧、チェック状態、メモ欄、前へ/次へ操作を表示する
4. フェーズ完了確認ダイアログ
   完了条件を満たしているか確認し、次フェーズへ進める
5. 完了サマリー画面
   実施結果、所要時間、メモ概要、完了日時を表示する
6. 履歴画面
   過去の完了セッションを参照できる

### 8.2 UI の基本原則

- 1 画面 1 目的を意識し、判断に迷わせない
- 「今どこか」「次に何をするか」を大きく表示する
- 必須確認項目は視覚的に区別する
- 誤操作を防ぐため、フェーズ移動時は確認を入れる
- 長文手順書でも読みやすい余白と階層を保つ

## 9. データ設計

### 9.1 手順書データモデル

初期版では JSON を採用し、以下のような構造を想定する。

```json
{
  "id": "sample-proc-001",
  "title": "機器セットアップ手順",
  "version": "1.0.0",
  "description": "新規端末の初期設定を行う手順",
  "estimatedMinutes": 30,
  "phases": [
    {
      "id": "phase-1",
      "title": "事前確認",
      "description": "必要物品と対象端末の状態を確認する",
      "completionCondition": "必須手順がすべて完了していること",
      "steps": [
        {
          "id": "step-1",
          "title": "電源状態を確認する",
          "instruction": "対象端末の電源が OFF であることを確認する",
          "required": true,
          "caution": "通電中に配線変更しないこと"
        }
      ]
    }
  ]
}
```

## 10. システム構成案

### 10.1 アーキテクチャ方針

小さなローカルアプリであり、かつマルチプラットフォーム対応が必要なため、初期実装では以下を第一候補とする。

- UI: React + TypeScript
- デスクトップアプリ基盤: Tauri
- ローカル保存: JSON ファイル、または SQLite

### 10.2 採用理由

- Tauri は軽量で、ローカルデスクトップアプリに向く
- React + TypeScript は UI 実装と保守がしやすい
- 手順書データは JSON で始めると扱いやすい
- セッション履歴が増える場合は SQLite に拡張しやすい

### 10.3 初期方針

- 手順書の定義ファイルは JSON
- 進捗保存はまずローカル JSON で開始する
- 履歴や検索要件が増えた場合は SQLite へ移行する

## 14. 現時点の提案

まずは「既存の手順書を JSON 化し、それをウィザード形式で安全に実行できるローカルデスクトップアプリ」を MVP として定義するのがよい。

この方針であれば、仕様が過度に広がらず、以下の価値を早く確認できる。

- 実際に作業ガイドとして使いやすいか
- フェーズ確認の体験が妥当か
- 現場で必要な記録粒度が十分か
- 手順書データの持ち方が実運用に耐えるか
